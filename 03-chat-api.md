# 模块说明：Chat API

`POST /api/chat` 是当前产品的主链路。它负责把用户输入、市场语境、平台风格和收藏样本整理成一轮可流式返回的回答。

## 请求输入

当前接口接收：

- `messages: UIMessage[]`
- `sessionId?: string`

前台不再显式传简繁模式。服务端会根据最新一条用户消息自动判断市场。

## 服务端处理流程

### 1. 取最新用户问题

从 `messages` 中反向找到最后一条 `role = user` 的文本内容，作为本轮判断依据。

### 2. 自动判断市场

根据关键词判断当前问题更接近：

- `CN`：内地语境
- `HK`：香港语境

这个结果会影响：

- system prompt 中的模式注释
- 收藏样本检索范围
- 本轮聊天写回数据库时的 `region`

### 3. 召回收藏样本

将当前问题转成 deterministic embedding，然后调用：

- `match_favorite_chat_memories`

只在当前市场下召回收藏样本，避免把内地样本和香港样本混在一起。

### 4. 拼装系统提示词

当前 prompt 由三部分组成：

1. 基础角色设定
2. 模式注释
3. 平台风格补充与 RAG 上下文

模式注释现在是显式拼接的：

- 简体模式：使用内地医美专业术语
- 繁体模式：使用 HK/TW 医美术语

### 5. 流式调用模型

通过 Vercel AI SDK 的 `streamText` 返回流式回答。

当前之所以坚持流式，不是为了炫技，而是因为这个产品的主要场景就是快速打样。  
如果第一屏迟迟不出字，体验会非常差。

### 6. 回答完成后写回数据库

流式完成后，把本轮问答落到 `chat_memories`，作为后续可能进入收藏池的原始素材。

## 降级策略

接口里有几处默认降级：

- Supabase RPC 失败：直接无 RAG 回答
- 入库失败：只打日志，不影响本轮输出
- 没有收藏样本：走纯 prompt 生成

这保证了数据库异常不会把主链路拖死。

## 响应头

当前响应头会带一些调试信息：

- `x-mei-session-id`
- `x-mei-market`
- `x-mei-rag-count`
- `x-mei-provider-ready`

这些头主要用于本地调试和排查，不是产品对外能力的一部分。
