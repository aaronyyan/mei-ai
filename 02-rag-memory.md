# 模块说明：轻量 RAG 与收藏问答库

Mei AI 这套 RAG 不是百科型知识库，也不是全量聊天记忆。它更像一个“团队风格样本库”。

## 目标

核心目标只有一个：让系统在生成新文案时，参考过去已经被认可的回答，尽量贴近团队自己的措辞和节奏。

这里“被认可”的标准，不是聊天发生过，而是用户明确点过“满意”。

## 数据来源

每轮聊天完成后，系统会把以下内容写入 `chat_memories`：

- `user_prompt`
- `assistant_response`
- `summary`
- `embedding`
- `region`
- `metadata`

只有当用户把某条回答标记为“满意”时，该条记录才会变成 `is_favorite = true`，进入召回范围。

## 为什么只召回 favorite

原因很简单：普通聊天里有大量草稿、试错和低质量表达，这些内容不适合反复喂给模型。  
如果把所有历史都召回，提示词只会越来越脏。

所以当前策略是：

- 自动沉淀所有问答
- 人工或业务侧只把真正满意的回答标记为 favorite
- RAG 只从 favorite 里找近邻

这能把样本量控制在小而精的范围内。

## 向量方案

当前使用的是本地 deterministic embedding，不依赖外部 embedding API。

处理流程如下：

1. 文本标准化
2. 英文词元切片
3. 中英文字符片段切片
4. hash 到 256 维向量
5. 向量归一化后写入 pgvector

这套方案的优点是稳定、便宜、依赖少。缺点也明确：语义能力有限，更适合做第一版风格召回，而不是复杂知识检索。

## 检索方式

当前通过 Supabase RPC `match_favorite_chat_memories` 做召回。

输入：

- 当前问题生成的查询向量
- `match_count`
- `match_region`

输出：

- 样本原始问题
- 样本回答
- 相似度
- metadata

服务端会把这些样本重新整理成 RAG 上下文，拼进系统提示词。

## RAG 在这个产品里真正起作用的地方

它最适合影响的是风格，不是事实知识。

例如：

- 句子是更短还是更舒展
- 是否偏向项目说明还是审美表达
- emoji 用得多还是少
- 标题更克制还是更像平台笔记

它不适合替代项目知识库，也不应该被当成医学事实来源。

## 当前局限

- 收藏样本少的时候，RAG 作用有限
- deterministic embedding 的召回质量有上限
- 现在更像“文风记忆”，不是“知识记忆”

但这不影响它在当前阶段有价值。对于医美文案这种强风格任务，小规模高质量样本库本身就已经能明显提升一致性。
